- **本次目标**：实现零服务器局域网直连

### 背景分析：为什么连不上

y-webrtc 依赖三层服务，三层全被 GFW 封锁：

| 环节 | 域名 | GFW |
|------|------|:--:|
| 信令 | `signaling.yjs.dev` | 🔴 墙 |
| 信令备选 | `*.herokuapp.com` | 🔴 墙 |
| STUN | `stun.l.google.com` | 🔴 墙 |
| TURN | 未配置 | ❌ 缺失 |

局域网同 WiFi 下其实不需要 STUN/TURN（mDNS 广播局域网 IP 即可直连），但**信令必须交换 SDP**，信令服务器被墙意味着 SDP 永远交换不了，局域网直连也无法建立。

### 实现方案

基于 `maiermic/serverless-webrtc` 的手动信令思路，写了一个自定义 Yjs Provider：

**操作流程**：
```
User A (发起方)                    User B (接受方)
  │                                  │
  ├─ 点击「发起连接」                │
  ├─ 自动生成 SDP offer               │
  ├─ 复制 base64 文本                 │
  ├─ 微信发给 B ──────────────────→  │
  │                                  ├─ 粘贴文本
  │                                  ├─ 自动生成 answer
  │                                  ├─ 复制 answer 文本
  │  ←────────────────────── 微信     │
  ├─ 粘贴 answer                     │
  │                                  │
  └─── WebRTC DataChannel 建立 ──────┘
         │
    Yjs CRDT 同步 (mDNS 局域网直连)
```

### 技术决策（ADR-013）
- **决策**：手动信令 SDP 打包为 base64(JSON)，包含 roomKey 防止同 WiFi 下串线
- **原因**：SDP 原始文本太长（~2-5KB），base64 紧凑；roomKey 做简单房间隔离
- **影响**：connectionId 用于追踪连接，roomKey 用于匹配双方

### 技术决策（ADR-014）
- **决策**：纯局域网模式不配置 iceServers（空数组）
- **原因**：同 WiFi 下 mDNS 直连无需 STUN/TURN，不配置 iceServers 避免浏览器去连被墙的服务器
- **影响**：仅限同 WiFi 使用，跨网络需后续加 TURN

### 连接类型支持表

| 模式 | 信令 | STUN | TURN | 适用场景 |
|------|------|------|------|---------|
| WebRTC（旧） | signaling.yjs.dev | Google STUN | 无 | 挂梯子可用 |
| 局域网直连（新） | 手动复制粘贴 | mDNS 直连 | 无 | 同 WiFi 零延迟 |

### Session 20 新发现 Bug

#### Bug #7 🔴 更新回声循环

**文件**：`src/lib/manual-signaling.ts:322`

**根因**：`channel.onmessage` 中 `Y.applyUpdate(doc, update)` 不带 origin 参数。`attachDocSync` 用 `origin === provider` 过滤本地更新——但远程到达的更新 origin 是 null/undefined，过滤失败。每条远程更新被原样发回。

**修复**：`Y.applyUpdate(this.doc, update, this)` — 将 provider 作为 origin 传入，使过滤器正确跳过远程到达的更新。

#### Bug #8 🟡 LAN 模式 peerStatus 页面指示器冻结

**文件**：`src/hooks/useLanSync.ts:161`

**根因**：`if (!currentPeer || currentPeer.userId !== key)` 守卫在首次设置后永久阻断后续更新。好友切换页面时 `currentPageIndex` 变化，但守卫不触发。

**修复**：加入 `currentPeer.currentPageIndex !== data.currentPageIndex` 守卫条件，页面切换时同步更新。

#### Bug #9 🟡 每次 Y.Doc 变化都重建 friendNotebook

**文件**：`src/hooks/useYjsSync.ts:148-176`

**根因**：observer handler 在 notebooks Map 的**任何**深度变化时触发，包括操作自己的笔记本。每次自己的操作都完整重读好友数据并调用 `setFriendNotebook`——即使好友数据没变。

补丁：`useLanSync.ts` 已有 `prevFriendRef` snapshot 比较（前次提交中），但 `useYjsSync.ts` 缺少同样的守卫。

**修复**：`useYjsSync.ts` 加入 `prevFriendRef` JSON snapshot 比较，只在好友数据实际变化时更新 Zustand。

#### Bug #10 — 确认为设计正确，不是 bug

**文件**：`src/hooks/useManualSync.ts:82-93`

`useEffect(cleanup, [])` 的空依赖是有意为之——`roomKey` 不变（只读 presetKey 或用户一次性输入），cleanup 只需在组件卸载时运行。加入 `roomKey` 依赖反而会导致重建期间销毁活跃的 provider。

## Session 19 续 — Bug 修复回合

### Bug #11: `{} is not iterable` — `useManualSync.ts` 监听器解构错误

- **根因**：`lib0` 的 `Observable.emit(name, args)` 使用 `f(...args)` 展开参数数组
  - `emit('state', [stateObj])` → 实际调用 `f(stateObj)` 而非 `f([stateObj])`
  - 监听器写 `([s]: [ConnectionState]) => ...` 期望数组解构 → 对普通对象解构 → crash
- **修复**：改为 `(s: ConnectionState) => { setState({ ...s }) }` （直接接收对象）
- **文件**：`src/hooks/useManualSync.ts`

### 全局错误捕获

- `index.html` 内联 `<script>` 在所有 JS 之前安装 `window.onerror` + `unhandledrejection`
- 底部紫色版本标签条显示 commit hash
- `main.tsx` 作为 fallback 补充 React 侧全局捕获

### 房间入口流程重构

- **旧流程**：先选创建/加入 → 再选在线/LAN → 房间码只读
- **新流程**：先选在线/LAN → 再输入房间码 + 创建/加入（3-step 状态机）
- **文件**：`src/components/room/RoomGate.tsx` 重写 → 删除 CreateRoom.tsx / JoinRoom.tsx
- **ManualConnect 增强**：`defaultRole` prop 预选角色 + 移除 `readOnly`

### Bug #12: WebRTC "wrong state: stable" + 电脑两个标签页独立房主

- **"wrong state: stable"** (手机+平板 LAN)：疑似双击并发 `createOffer` → 第二个调用关闭第一个 PC → 状态塌陷
  - 修复：所有 handler 加 `if (loading) return` 防重入守卫 + 按钮 `disabled={... || loading}`
  
- **电脑两个标签页独立房主** (LAN 模式) 🔑 **本次核心修复**：
  - **根因**：ManualConnect 完成 SDP 交换后 `onConnected` → ManualConnect 卸载 → `useManualSync` cleanup 调用 `provider.destroy()` → **关闭 WebRTC DataChannel**
  - `LanRoomApp` → `useLanSync` 创建的是空壳 `provider`（stub），没有任何网络同步能力
  - 结果：两个标签页各有一个本地 `Y.Doc`，但 DataChannel 已死，文档变更永不互通
  - **修复**：
    1. `useManualSync`：`synced=true` 时不销毁 provider，转移到模块级变量 `_lanProvider`
    2. `useLanSync`：检测 `_lanProvider` → 通过 `attachDocSync(provider)` 重建 `doc → DataChannel` 桥接

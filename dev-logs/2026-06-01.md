## Session 19 — 局域网直连模式（手动信令 WebRTC）

### 会话上下文
- **前置会话**：Session 18 GoodNotes 工具栏 + 折叠按钮修复
- **用户需求**：解决国内跨设备连接问题，先做好局域网（同 WiFi）P2P
- **本次目标**：实现零服务器局域网直连

### 背景分析：为什么连不上

y-webrtc 依赖三层服务，三层全被 GFW 封锁：

| 环节 | 域名 | GFW |
|------|------|:--:|
| 信令 | `signaling.yjs.dev` | 🔴 墙 |
| 信令备选 | `*.herokuapp.com` | 🔴 墙 |
| STUN | `stun.l.google.com` | 🔴 墙 |
| TURN | 未配置 | ❌ 缺失 |

局域网同 WiFi 下其实不需要 STUN/TURN（mDNS 广播局域网 IP 即可直连），但**信令必须交换 SDP**，信令服务器被墙意味着 SDP 永远交换不了，局域网直连也无法建立。

### 搜索参考项目

| 仓库 | ⭐ | 思路 |
|------|:--:|------|
| [maiermic/serverless-webrtc](https://github.com/maiermic/serverless-webrtc) | - | 纯手动复制粘贴 SDP，零服务器 |
| [cjb/serverless-webrtc](https://github.com/cjb/serverless-webrtc) | - | 同上，Bootstrap 模态框引导 |
| [svarunan/serverless-webrtc](https://github.com/svarunan/serverless-webrtc) | 95 | 同上，MIT 协议 |
| [WinstonFassett/y-webrtc-trystero](https://github.com/WinstonFassett/y-webrtc-trystero) | - | Yjs + Trystero（BitTorrent 信令），国内大概率被墙 |
| [dmotz/trystero](https://github.com/dmotz/trystero) | 2560+ | 去中心化 WebRTC 信令库 |

### 实现方案

基于 `maiermic/serverless-webrtc` 的手动信令思路，写了一个自定义 Yjs Provider：

**操作流程**：
```
User A (发起方)                    User B (接受方)
  │                                  │
  ├─ 点击「发起连接」                │
  ├─ 自动生成 SDP offer               │
  ├─ 复制 base64 文本                 │
  ├─ 微信发给 B ──────────────────→  │
  │                                  ├─ 粘贴文本
  │                                  ├─ 自动生成 answer
  │                                  ├─ 复制 answer 文本
  │  ←────────────────────── 微信     │
  ├─ 粘贴 answer                     │
  │                                  │
  └─── WebRTC DataChannel 建立 ──────┘
         │
    Yjs CRDT 同步 (mDNS 局域网直连)
```

### 技术决策（ADR-013）
- **决策**：手动信令 SDP 打包为 base64(JSON)，包含 roomKey 防止同 WiFi 下串线
- **原因**：SDP 原始文本太长（~2-5KB），base64 紧凑；roomKey 做简单房间隔离
- **影响**：connectionId 用于追踪连接，roomKey 用于匹配双方

### 技术决策（ADR-014）
- **决策**：纯局域网模式不配置 iceServers（空数组）
- **原因**：同 WiFi 下 mDNS 直连无需 STUN/TURN，不配置 iceServers 避免浏览器去连被墙的服务器
- **影响**：仅限同 WiFi 使用，跨网络需后续加 TURN

### 连接类型支持表

| 模式 | 信令 | STUN | TURN | 适用场景 |
|------|------|------|------|---------|
| WebRTC（旧） | signaling.yjs.dev | Google STUN | 无 | 挂梯子可用 |
| 局域网直连（新） | 手动复制粘贴 | mDNS 直连 | 无 | 同 WiFi 零延迟 |

### 初版测试问题

初版部署后立即发现两个问题：

1. **`doc 未初始化`**：`ensureProvider` 中 `docRef.current` 为 null → 用户点击「发起连接」时报错。根因：React 状态更新是异步的，`setDoc(docRef.current)` 后组件尚未重渲染，`ensureProvider` 已通过 `docRef.current` 读到 null。修复：在 `handleCreate` 等入口先用 `ensureDoc()` 同步写 ref.current。

2. **`doc 使用 useState 导致闭包过时`**：`ensureProvider` 的 `useCallback` 依赖 `doc`（useState 值），但异步操作中闭包捕获的 `doc` 可能已是旧值。修复：将 `doc` 参数改为 `MutableRefObject<Y.Doc | null>`，调用方直接写 `ref.current`，不依赖 React 重渲染。

### 审查中新发现 Bug

#### Bug #7 🔴 更新回声循环

**文件**：`src/lib/manual-signaling.ts:322`

**根因**：`channel.onmessage` 中 `Y.applyUpdate(doc, update)` 不带 origin 参数。`attachDocSync` 用 `origin === provider` 过滤本地更新——但远程到达的更新 origin 是 null/undefined，过滤失败。每条远程更新被原样发回 → 回声循环 → DataChannel 被消息淹没。

**修复**：`Y.applyUpdate(this.doc, update, this)` — 将 provider 作为 origin 传入，使过滤器正确跳过远程到达的更新。

#### Bug #8 🟡 LAN 模式 peerStatus 页面指示器冻结

**文件**：`src/hooks/useLanSync.ts:161`

**根因**：`if (!currentPeer || currentPeer.userId !== key)` 守卫在首次设置后永久阻断后续更新。好友切换页面时 `currentPageIndex` 变化，但守卫不触发更新。

**修复**：加入 `currentPeer.currentPageIndex !== data.currentPageIndex` 守卫条件，页面切换时同步更新 peerStatus。

#### Bug #9 🟡 每次 Y.Doc 变化都重建 friendNotebook

**文件**：`src/hooks/useYjsSync.ts:148-176`

**根因**：observer handler 在 notebooks Map 的**任何**深度变化时触发，包括操作自己的笔记本。每次自己的操作都完整重读好友数据并调用 `setFriendNotebook`——即使好友数据没变。

**修复**：`useYjsSync.ts` 加入 `prevFriendRef` JSON snapshot 比较，只在好友数据实际变化时更新 Zustand（`useLanSync.ts` 已有此守卫）。

#### Bug #10 — 确认为设计正确，不是 bug

**文件**：`src/hooks/useManualSync.ts:82-93`

`useEffect(cleanup, [])` 的空依赖是有意为之——`roomKey` 不变（presetKey 或用户一次性输入），cleanup 只需在组件卸载时运行。加入 `roomKey` 依赖反而会导致重建期间销毁活跃的 provider。

## Session 21 — Bug #11 定位：`{} is not iterable` 崩溃

### 现象

用户点击「局域网直连」标签页 → 立即崩溃。已部署的 Error Boundary (EB) 未捕获任何错误。

### 排查过程

**第一步：Error Boundary 未生效的原因**

React Error Boundaries (`componentDidCatch` / `getDerivedStateFromError`) 只能捕获**渲染阶段**的错误：
- ✅ 组件 `render()` 中的错误
- ❌ 事件处理器中的错误（如 `onClick`）
- ❌ `Promise` rejection（如 `async/await`）
- ❌ 模块初始化错误（顶层 `import` 执行）
- ❌ `setTimeout` / `requestAnimationFrame` 回调中的错误

Bug #11 的错误发生在事件处理器或 Promise 回调中，EB 无法捕获。

**第二步：全局错误捕获**

在 `index.html` 中于所有 JS 加载之前安装全局处理器：

```javascript
// 捕获同步错误
window.onerror = function(msg, file, line, col, error) { ... }

// 捕获 Promise 错误
window.addEventListener('unhandledrejection', function(event) { ... })
```

在 `main.tsx` 中补充 React 侧的 fallback（某些 React 18 错误不会冒泡到 window.onerror）。

同时添加 `showErrorBox()` 函数：红色全宽顶栏显示错误消息 + 堆栈，异常醒目。

**第三步：从堆栈定位根因**

用户反馈的堆栈指向 `useManualSync.ts` → `emit('state', [this._state])` → listener 中的数组解构。

### 根因：lib0 Observable.emit 的参数展开

`lib0` 的 `Observable.emit(name, args)` 实现：

```typescript
emit(name: string, args: any[]) {
  // 内部调用: f(...args)
  // 即 emit('state', [stateObj]) → f(stateObj) 而非 f([stateObj])
}
```

原监听器代码：
```typescript
provider.on('state', ([s]: [ConnectionState]) => {
  setState({ ...s })  // 💥 对普通对象解构 → "object is not iterable"
})
```

`emit('state', [stateObj])` 将数组展开为参数，所以监听器实际收到的是 `stateObj`（一个普通对象），而非 `[stateObj]`（数组）。数组解构 `([s])` 期望参数是可迭代的数组 → crash。

### 修复

```typescript
// 修复后：直接接收对象
provider.on('state', (s: ConnectionState) => {
  setState({ ...s })
})

provider.on('synced', (data: { synced: boolean }) => {
  setSynced(data.synced)
})
```

两个监听器都受影响：
- `state` 监听器：`([s]: [ConnectionState])` → `(s: ConnectionState)`
- `synced` 监听器：`([data]: [{ synced: boolean }])` → `(data: { synced: boolean })`

**教训**：lib0 的 `emit(name, args)` 与 DOM 的 `dispatchEvent(event)` 不同——前者展开 args 数组为独立参数，后者传递单个 event 对象。

### 全局错误捕获体系（最终方案）

| 层级 | 位置 | 机制 | 捕获范围 |
|------|------|------|---------|
| 1 | `index.html` 内联 `<script>` | `window.onerror` | 同步错误（所有） |
| 2 | `index.html` 内联 `<script>` | `unhandledrejection` | Promise 错误 |
| 3 | `main.tsx` | `window.addEventListener('error')` | React 18 不冒泡的错误 |
| 4 | `main.tsx` | `window.addEventListener('unhandledrejection')` | fallback |
| 5 | ManualConnect.tsx | EB (Error Boundary) | 渲染阶段错误 |

## Session 22 — 房间入口流程重构

### 用户反馈

> "修改房间码为可以修改与复制粘贴吧，这边每个标签页进入时，要通过创建新房间才能选择局域网直连，此时两个标签页都会自动生成一个新的房间号，逻辑很混乱。你改成选择连接方式-在线/局域网，然后创建/加入吧"

旧流程问题：
1. 逻辑颠倒：先选「创建/加入」→ 再选「在线/LAN」
2. 每次「创建」自动生成不同随机码 → 两个标签页各得不同码，无法互通
3. 房间码只读不可编辑 → 无法手动匹配

### 新设计：三步骤状态机

```
选择连接方式                  设置房间码                      局域网直连
+------------------+       +------------------------+       +-------------------+
|  在线连接         |  ──→  | 房间码 [可编辑/自动大写] |       | ManualConnect     |
|  局域网直连       |       | [生成] [复制链接]        |  ──→  | (复用现有组件)     |
+------------------+       | [创建房间] [加入房间]    |       +-------------------+
        ↑                  +------------------------+             |
        |                     |            |                      |
        +--- 返回 ←-----------+  在线: onJoin()    LAN: onLanConnect()
                                        进入房间         进入房间
```

### 状态变量

| 状态 | 类型 | 说明 |
|------|------|------|
| `step` | `'choose-mode' \| 'room-entry' \| 'lan-connect'` | 当前步骤 |
| `connMode` | `'online' \| 'lan'` | 连接方式 |
| `roomCode` | `string` | 可编辑，初始为空，maxLength=10，autoUpperCase |
| `action` | `'create' \| 'join' \| null` | 创建还是加入 |
| `userId` | `string` | `useState(() => generateUserId())` |

## Session 23 — Bug #12: WebRTC 连接失败（两个子问题）

### 测试场景

用户测试反馈两个问题：
1. **手机+平板** LAN 连接：发起方粘贴回传码时报 `Failed to set remote answer sdp: Called in wrong state: stable`
2. **电脑两个标签页** LAN 连接：两个标签页都成功进入笔记本界面，但各自为独立房主，无法看到对方的笔记本内容

### 子问题 A: WebRTC "wrong state: stable"

**WebRTC 信令状态机**：
```
stable → (setLocalDescription offer) → have-local-offer → (setRemoteDescription answer) → stable
```

`acceptAnswer` 调用 `pc.setRemoteDescription(answer)` 要求 PC 处于 `have-local-offer` 状态。如果处于 `stable`，说明：
- `setLocalDescription` 没被调用过
- 或 PC 被重建了（`createPeerConnection()` 会先 close 旧 PC → 新建 → 回到 `stable`）

**推断根因**：移动端触摸双击 → 两个并发 `createOffer` 调用 → 第二个调用触发 `createPeerConnection()` → `this.pc.close()` 关闭了第一个正在等待 ICE 的 PC → 第一个调用结束时 PC 已死 → 用户粘贴 answer 时 PC 处于 stable（因为被重建了）

**修复**：
- 所有 WebRTC handler 加 `if (loading) return` 防重入守卫
- 所有操作按钮 `disabled` 增加 `|| loading` 条件
- 修改文件：`ManualConnect.tsx` — `handleCreate`、`handleAcceptAnswer`、`handleAcceptOffer`

### 子问题 B: 两个标签页独立房主（核心修复）🔑

**现象**：电脑两个标签页完成 LAN SDP 交换 → 都进入笔记本界面 → 各自独立，修改不互通。

**根因分析**（组件生命周期问题）：

```
ManualConnect 完成连接 (DataChannel opened, synced=true)
  → useEffect 检测 synced → setTimeout 800ms → onConnected(doc)
    → RoomGate.onLanConnect(doc, userId, roomCode)
      → App.setLan({ doc, userId })
        → App 重渲染 → RoomGate 卸载
          → ManualConnect 卸载
            → useManualSync cleanup 执行
              → provider.destroy() 💥 关闭 DataChannel & PeerConnection
                → LanRoomApp 渲染
                  → useLanSync 创建空壳 provider (stub)
                    → 两个标签页各有本地 Y.Doc，但 WebRTC 通道已死
```

关键发现：`useLanSync` 创建的 provider 是一个**纯空壳 stub**：

```typescript
// useLanSync.ts — 原代码
function createLocalSync(doc: Y.Doc): YjsSync {
  return {
    doc,
    provider: {
      awareness: { setLocalState: () => {}, getStates: () => new Map(), ... },
      on: () => {},
      off: () => {},
      disconnect: () => {},
    },
    destroy: () => {},
  }
}
```

所有网络方法都是 `() => {}`。真正的 WebRTC DataChannel 在 ManualConnect 卸载时已被销毁。Y.Doc 虽然通过 `onConnected(doc)` 传递给了 LanRoomApp，但没有任何传输层把它同步到对端。

**修复**：provider 生命周期跨组件存活

1. **`useManualSync.ts`**：
   - 连接建立后（`synced=true`），cleanup **不销毁** provider，而是 detach 旧 `doc→DC` 桥接后将 provider 引用转移到模块级变量 `_lanProvider`
   - 导出 `getLanProvider()` 和 `clearLanProvider()` 供 useLanSync 接管

2. **`useLanSync.ts`**：
   - 初始化时通过 `getLanProvider()` 检测是否有存活 provider
   - 如果有 → `attachDocSync(provider)` 重建 `doc → DataChannel` 桥接
   - 如果无 → fallback 到空壳 stub（理论不应发生）

**数据流（修复后）**：
```
User draws on canvas
  → Zustand store 更新
    → useLanSync writes to Y.Doc (doc.getMap('notebooks')...)
      → Y.Doc fires 'update' event
        → attachDocSync handler catches it
          → syncStep2 sends update via DataChannel
            → Remote peer receives onmessage
              → Y.applyUpdate(remoteDoc, update, provider)
                → Remote Y.Doc fires observer
                  → useLanSync reads from doc
                    → Zustand setFriendNotebook
                      → UI re-renders
```

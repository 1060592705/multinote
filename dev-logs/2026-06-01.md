### 背景分析：为什么连不上

y-webrtc 依赖三层服务，三层全被 GFW 封锁：

| 环节 | 域名 | GFW |
|------|------|:--:|
| 信令 | `signaling.yjs.dev` | 🔴 墙 |
| 信令备选 | `*.herokuapp.com` | 🔴 墙 |
| STUN | `stun.l.google.com` | 🔴 墙 |
| TURN | 未配置 | ❌ 缺失 |

局域网同 WiFi 下其实不需要 STUN/TURN（mDNS 广播局域网 IP 即可直连），但**信令必须交换 SDP**，信令服务器被墙意味着 SDP 永远交换不了，局域网直连也无法建立。

### 实现方案

基于 `maiermic/serverless-webrtc` 的手动信令思路，写了一个自定义 Yjs Provider：

**操作流程**：
```
User A (发起方)                    User B (接受方)
  │                                  │
  ├─ 点击「发起连接」                │
  ├─ 自动生成 SDP offer               │
  ├─ 复制 base64 文本                 │
  ├─ 微信发给 B ──────────────────→  │
  │                                  ├─ 粘贴文本
  │                                  ├─ 自动生成 answer
  │                                  ├─ 复制 answer 文本
  │  ←────────────────────── 微信     │
  ├─ 粘贴 answer                     │
  │                                  │
  └─── WebRTC DataChannel 建立 ──────┘
         │
    Yjs CRDT 同步 (mDNS 局域网直连)
```

### 技术决策（ADR-013）
- **决策**：手动信令 SDP 打包为 base64(JSON)，包含 roomKey 防止同 WiFi 下串线
- **原因**：SDP 原始文本太长（~2-5KB），base64 紧凑；roomKey 做简单房间隔离
- **影响**：connectionId 用于追踪连接，roomKey 用于匹配双方

### 技术决策（ADR-014）
- **决策**：纯局域网模式不配置 iceServers（空数组）
- **原因**：同 WiFi 下 mDNS 直连无需 STUN/TURN，不配置 iceServers 避免浏览器去连被墙的服务器
- **影响**：仅限同 WiFi 使用，跨网络需后续加 TURN
